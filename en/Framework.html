SHS Modular Framework Guide

This guide explains how to configure the Shadow Haven Studios modular framework for Arma Reforger. It covers SHS item types, policy-driven storage slots, placement offsets, loadout-area routing, compatibility aliases, and common troubleshooting steps.

Contents 1. Framework Overview 2. File and Class Map 3. Item Types and Families 4. Storage and Slot Setup 5. Visual Placement Offsets 6. Loadout Area Routing 7. How-To Recipes 8. Troubleshooting 9. Authoring Checklist

1. Framework Overview

The SHS Modular Framework is a policy-driven inventory routing framework. Instead of relying only on vanilla item type checks, the framework classifies items into SHS-owned families, scans actual configured storage slots, scores valid destinations, and routes the item into the best matching slot.

Core goals

Policy-driven routing Items are routed based on configured slot policy, item family, exact prefab rules, slot category, and occupancy group.
Compatibility-safe typing SHS uses its own ECommonItemType values and compatibility aliases so the framework does not depend on another mod’s implementation.
Physical-space protection Multiple slot variants can share one m_SHS_OccupancyGroup, preventing duplicate use of the same physical space.
Visual placement control SHS_MagPlacementComponent applies local offsets and rotations after insertion so items display correctly on placards, belts, radios, pouches, or modular gear.
Core rule: Configure the slot, not the insertion path. The router should discover the correct destination from the storage and slot configuration.

2. File and Class Map

File / Class Purpose
Scripts/Game/SHS ItemTypes MSC.c Defines SHS-owned ECommonItemType values, ammo categories, utility categories, and the auto-routed sentinel range.
Scripts/Game/Vissible item types.c Handles first-person visibility and hidden item behavior for auto-routed types and extra manual hide types.
SHS_EquipmentStorageSlot.c Main policy slot logic. Handles category matching, exact prefab filtering, occupancy groups, family rebound, and visual placement correction.
SHS_EquipmentStorageComponent.c Storage component policy flag and component-level slot validation.
SHS_PolicyDrivenStorageSlot.c Base class for specialized SHS policy slots such as radio, patch, grenade, holster, and accessory slots.
SHS_ItemClassificationService.c Central item classification service. Resolves item family, type normalization, resource checks, and auto-routed detection.
SHS_StorageRouteResolver.c Collects candidate slots, scores them, chooses the best slot, and executes insert or move routes.
SHS_MagGrenadePolicyRouter.c Routes magazines, grenades, smoke, and utility items through configured storage slots.
SHS_MagPlacementComponent.c Applies per-slot visual offset and rotation data by prefab, category, and placement ID.
SHS_ItemTypeOverrideComponent.c Optional prefab-level type override when external or vanilla common item types should not be trusted.
SHS_ExternalCompatibilityAliases.c Normalizes external framework type names into SHS-owned categories.
Area Routing Types.c Registers LoadoutAreaType values to SHS family behavior.
SHS_ClothAreaExtensionComponent.c Adds extra SHS loadout area types to cloth or gear prefabs without replacing the engine’s canonical area.

Routing flow

  1. The item enters inventory logic.
  2. SHS_ItemClassificationService classifies the item/resource.
  3. The resolver scans managed storage components and their slots.
  4. Each slot validates the item against category, family, prefab, and occupancy rules.
  5. The best candidate is selected based on route priority and match quality.
  6. The item is inserted or moved into the selected slot.
  7. Post-attach placement correction applies offset and rotation data.

3. Item Types and Families

SHS groups items into families so that slots can accept behavior categories instead of relying only on raw vanilla common item types.

SHS item families

AMMO SMOKE GRENADE RADIO PATCH POUCH ATAK AUTO

Important SHS common item types

SHS_ATAK SHS_RADIO SHS_PATCH SHS_GRENADE SHS_SMOKE_GRENADE SHS_PISTOL_AMMO SHS_RIFLE_AMMO SHS_MARKSMAN_AMMO SHS_MG_AMMO SHS_AK_AMMO SHS_GL_AMMO SHS_Pouch SHS_EAR_LEFT SHS_EAR_RIGHT SHS_EAR_LEFT_REAR_MOUNT SHS_EAR_Right_REAR_MOUNT SHS_NightVision SHS_RailLiteRight SHS_RailLiteLeft SHS_Waist SHS_R_Leg SHS_L_Leg SHS_Sling SHS_Mask
Auto-routed range: Item types placed between SHS_AUTO_ROUTED_BEGIN and SHS_AUTO_ROUTED_END are treated as AUTO-routed gear. These types participate in SHS routing and are hidden in first person. Do not assign the sentinel markers themselves to prefabs.

Type resolution order

Source Use
Native / vanilla common type Used when the prefab already has a reliable ECommonItemType.
m_SHS_AdditionalCommonItemTypes Sidecar list for items that need additional SHS routing tags without breaking their original type.
SHS_ItemTypeOverrideComponent Prefab-level override when SHS should classify the item before trusting external or vanilla types.
SHS_ExternalCompatibilityAliases Converts external type names into SHS-owned types.

Recommended tagging rules

For SHS-owned items:
  Use SHS-owned ECommonItemType values directly
  or add them through m_SHS_AdditionalCommonItemTypes.

For external compatibility:
  Prefer SHS_ItemTypeOverrideComponent
  or use the additional common item type sidecar list.

For AUTO-routed helmet or gear accessories:
  Add the item type between SHS_AUTO_ROUTED_BEGIN and SHS_AUTO_ROUTED_END.
  Configure the destination slot's m_SHS_MagCategory to that exact SHS type.

4. Storage and Slot Setup

Primary SHS slot fields

Field Meaning
m_SHS_MagCategory Category gate. Set this to the SHS type the slot should accept, such as SHS_RIFLE_AMMO, SHS_RADIO, SHS_GRENADE, SHS_SMOKE_GRENADE, or an AUTO-routed gear type.
m_SHS_SlotID Human-readable routing and placement ID. This must match SHS_MagPlacementData.PlacementID when visual placement offsets are used.
m_SHS_ItemPrefab Exact prefab/resource filter. Leave blank for category fallback. Fill it when one exact item should win this slot.
m_SHS_OccupancyGroup Logical physical-space lock. Slots with the same occupancy group block each other so multiple variants cannot occupy the same visual cell.

Policy-driven equipment storage example

Storage owner prefab
└─ SHS_EquipmentStorageComponent
   ├─ m_bSHS_PolicyDriven = true
   ├─ Slot 0: SHS_EquipmentStorageSlot
   │  ├─ m_SHS_MagCategory    = SHS_RIFLE_AMMO
   │  ├─ m_SHS_SlotID         = Placard_Cell_01
   │  └─ m_SHS_OccupancyGroup = Placard_Cell_01
   │
   ├─ Slot 1: SHS_EquipmentStorageSlot
   │  ├─ m_SHS_MagCategory    = SHS_PISTOL_AMMO
   │  ├─ m_SHS_SlotID         = Belt_Cell_01
   │  └─ m_SHS_OccupancyGroup = Belt_Cell_01
   │
   └─ Slot 2: SHS_EquipmentStorageSlot
      ├─ m_SHS_MagCategory    = SHS_SMOKE_GRENADE
      ├─ m_SHS_SlotID         = Chest_Left_01
      └─ m_SHS_OccupancyGroup = Chest_Left_01

Special policy slots

Slot class Use case
SHS_RadioStorageSlot Radio pouch or radio cell. Accepts radio components and/or SHS_RADIO typed items.
SHS_NadeStorageSlot Explosive, frag, grenade, or grenade-pouch behavior.
SHS_PatchStorageSlot Patch, admin, callsign, or ID panel slot accepting SHS_PATCH.
SHS_HolsterStorageSlot Sidearm-specific holster behavior using configured weapon checks.
SHS_AccessorySlot Chest accessory slot with allowed type arrays for additional filtering.
Avoid mixed-family fallback: A NONE slot category is treated as legacy ammo fallback. Do not use NONE for radios, smoke, grenades, patches, or AUTO gear. Give utility slots an explicit SHS category.

5. Visual Placement Offsets

Use SHS_MagPlacementComponent on the storage owner when a stored item needs local visual adjustment after insertion.

Placement fields

Field Meaning
Prefab Exact item prefab/resource path. This is the highest-priority placement match. Leave blank for type or slot fallback.
MagType Fallback common item type, such as SHS_RIFLE_AMMO or SHS_SMOKE_GRENADE.
PlacementID Must match the slot’s m_SHS_SlotID.
Offset Local position offset applied after item attachment.
Angles Local rotation in Enfusion angle vector format.

Example placement setup

SHS_MagPlacementComponent.m_aPlacements:
  Entry 0:
    Prefab      = ""
    MagType     = "SHS_RIFLE_AMMO"
    PlacementID = "Placard_Cell_01"
    Offset      = "0.015 0.020 -0.005"
    Angles      = "0 90 0"

Slot:
  m_SHS_MagCategory    = SHS_RIFLE_AMMO
  m_SHS_SlotID         = Placard_Cell_01
  m_SHS_OccupancyGroup = Placard_Cell_01
Why PlacementID matters: the item is first attached by the engine, then SHS corrects the local transform. If PlacementID does not match the slot’s m_SHS_SlotID, the placement rule may not apply.

6. Loadout Area Routing

SHS loadout-area routing allows gear to declare additional valid areas without destroying its vanilla or engine-level area assignment.

Use SHS_ClothAreaExtensionComponent

Attach SHS_ClothAreaExtensionComponent to clothing or gear prefabs when the prefab needs to participate in additional SHS loadout areas. 

Example:
  Prefab: Tactical_Radio_Pouch.et

  Native area:
    LoadoutAreaType = VEST

  SHS extension:
    m_aAdditionalSHSAreas:
      - SHS_RADIO
      - SHS_POUCH

When to use additional areas

Situation Recommended setup
Gear should remain vanilla-compatible Keep the original vanilla loadout area and add SHS areas with SHS_ClothAreaExtensionComponent.
Gear needs SHS-only sorting Add one or more SHS loadout areas and route through the SHS family registry.
Gear is being ignored by SHS loadout routing Verify the SHS area exists in the registry and that the prefab has the extension component.

7. How-To Recipes

Recipe A: Add a rifle magazine placard cell

  1. Open the placard or vest prefab.
  2. Add or locate the SHS_EquipmentStorageComponent.
  3. Enable policy-driven behavior.
  4. Add a new SHS_EquipmentStorageSlot.
  5. Set m_SHS_MagCategory to SHS_RIFLE_AMMO.
  6. Set m_SHS_SlotID to a unique value, such as Placard_Rifle_01.
  7. Set m_SHS_OccupancyGroup to the same physical cell name.
  8. Add a placement entry matching Placard_Rifle_01.
  9. Test insertion in Workbench and verify the visual transform.
Slot:
  m_SHS_MagCategory    = SHS_RIFLE_AMMO
  m_SHS_SlotID         = Placard_Rifle_01
  m_SHS_OccupancyGroup = Placard_Rifle_01

Placement:
  MagType     = SHS_RIFLE_AMMO
  PlacementID = Placard_Rifle_01
  Offset      = "0 0 0"
  Angles      = "0 0 0"

Recipe B: Add a smoke grenade slot

  1. Create or locate the smoke pouch slot.
  2. Use SHS_EquipmentStorageSlot or a specialized grenade policy slot if required.
  3. Set m_SHS_MagCategory to SHS_SMOKE_GRENADE.
  4. Give the slot a unique m_SHS_SlotID.
  5. Give the slot a matching occupancy group.
  6. Add placement offset data if the visual item needs alignment.
Slot:
  m_SHS_MagCategory    = SHS_SMOKE_GRENADE
  m_SHS_SlotID         = Smoke_Left_01
  m_SHS_OccupancyGroup = Smoke_Left_01

Recipe C: Add a radio pouch

  1. Create the radio pouch slot.
  2. Use SHS_RadioStorageSlot when radio-specific validation is needed.
  3. Set the slot category to SHS_RADIO.
  4. Set a clear slot ID, such as Radio_Back_Left.
  5. Use occupancy grouping if alternate radio positions share the same space.
Slot:
  Class                = SHS_RadioStorageSlot
  m_SHS_MagCategory    = SHS_RADIO
  m_SHS_SlotID         = Radio_Back_Left
  m_SHS_OccupancyGroup = Radio_Back_Left

Recipe D: Add a patch slot

  1. Add a patch storage slot to the uniform, vest, helmet, or admin panel.
  2. Use SHS_PatchStorageSlot.
  3. Set the category to SHS_PATCH.
  4. Set the slot ID to the visual location.
Slot:
  Class                = SHS_PatchStorageSlot
  m_SHS_MagCategory    = SHS_PATCH
  m_SHS_SlotID         = Patch_Chest_Admin
  m_SHS_OccupancyGroup = Patch_Chest_Admin

Recipe E: Add AUTO-routed helmet accessory gear

  1. Create the new SHS item type inside the auto-routed range.
  2. Assign that type to the item prefab or add it through the additional type list.
  3. Create a destination slot on the helmet or gear platform.
  4. Set the slot’s m_SHS_MagCategory to the exact AUTO-routed type.
  5. Add placement data using the slot’s m_SHS_SlotID.
Item:
  AdditionalCommonItemTypes:
    - SHS_RailLiteRight

Slot:
  m_SHS_MagCategory    = SHS_RailLiteRight
  m_SHS_SlotID         = Helmet_Rail_Right_01
  m_SHS_OccupancyGroup = Helmet_Rail_Right_01

Placement:
  MagType     = SHS_RailLiteRight
  PlacementID = Helmet_Rail_Right_01
  Offset      = "0.02 0.01 0"
  Angles      = "0 0 0"

Recipe F: Use exact prefab priority

Use m_SHS_ItemPrefab when a specific item must prefer a specific slot before generic category fallback. 

Slot:
  m_SHS_MagCategory = SHS_RIFLE_AMMO
  m_SHS_ItemPrefab  = "{RESOURCE_ID}Prefabs/Items/Magazines/SHS_556_PMAG.et"
  m_SHS_SlotID      = Placard_PMAG_01
Exact prefab rules should be used sparingly. Use category routing for general behavior and exact prefab rules only when a specific visual model needs a specific position.

8. Troubleshooting

Issue Likely cause / fix
Radio goes into ammo cell The radio slot may be using NONE or a legacy fallback category. Set the slot to SHS_RADIO and use SHS_RadioStorageSlot if needed.
Smoke grenades go into the wrong row The smoke item may not have SHS_SMOKE_GRENADE, or the destination slot is using the wrong category. Check both item tagging and slot m_SHS_MagCategory.
SHS additional item types are ignored Verify the item actually exposes m_SHS_AdditionalCommonItemTypes to the classification service. If another framework overrides the base type, add SHS_ItemTypeOverrideComponent or a compatibility alias.
Item inserts but appears rotated or floating The item routed correctly, but placement data is missing or the PlacementID does not match the slot’s m_SHS_SlotID.
Two items occupy the same visual pouch The related slots do not share the same m_SHS_OccupancyGroup. Give all mutually exclusive variants the same occupancy group.
Patch does not route Confirm the patch item has SHS_PATCH and the slot uses SHS_PatchStorageSlot or an SHS slot with m_SHS_MagCategory = SHS_PATCH.
AUTO gear is visible in first person The item type may not be inside the auto-routed range or may not be included in the visible item hide logic. Verify the type placement and first-person hide configuration.
Loadout editor ignores custom area Add SHS_ClothAreaExtensionComponent and verify the area is registered in the SHS loadout-area family registry.

9. Authoring Checklist

For every SHS storage slot

  • Use an SHS policy slot class when special behavior is needed.
  • Set m_SHS_MagCategory to a real SHS category.
  • Do not use NONE for radios, patches, smoke, grenades, or AUTO gear.
  • Set a clear and unique m_SHS_SlotID.
  • Set m_SHS_OccupancyGroup for any physical cell that can have multiple variants.
  • Use m_SHS_ItemPrefab only when exact prefab priority is required.

For every routed item

  • Assign the correct SHS common item type.
  • Use m_SHS_AdditionalCommonItemTypes when the item needs extra routing tags.
  • Use SHS_ItemTypeOverrideComponent when vanilla or external types cannot be trusted.
  • Confirm compatibility aliases exist for external framework type names.

For every visual placement

  • Add SHS_MagPlacementComponent to the storage owner.
  • Match PlacementID to the slot’s m_SHS_SlotID.
  • Use exact prefab placement only when necessary.
  • Use type-based placement for reusable category behavior.
  • Test offset and angle values in Workbench.

For every custom loadout area

  • Keep the vanilla area when vanilla compatibility is needed.
  • Add SHS extra areas through SHS_ClothAreaExtensionComponent.
  • Verify the area is registered in SHS loadout routing types.
  • Test the item in the loadout editor and in-game inventory.
Best practice: Treat SHS categories, slot IDs, occupancy groups, and placement IDs as a single connected configuration. Most routing problems come from one of those values not matching.
