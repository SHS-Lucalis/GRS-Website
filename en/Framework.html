1. What this framework does

Policy-driven routing

Items are not thrown into the first vanilla-compatible slot. SHS checks the configured slot policy, item family, exact prefab rule, slot category, and occupancy group before accepting an insert.

Compatibility-safe item typing

SHS uses its own ECommonItemType values and normalizes external names through aliases instead of depending on another framework’s implementation.

Physical-space protection

Multiple visual variants can share one m_SHS_OccupancyGroup. Once one variant is filled, the other variants in that group are blocked.

Visual placement control

SHS_MagPlacementComponent applies local offset and angle data after the engine attaches the item, preventing the engine attach pass from wiping the transform.

Core concept: Configure the slot, not the item insertion path. The router scans actual storage slots, scores valid candidates, and inserts into the best configured destination.

Routing flow

Item enters inventory logic. The modded inventory manager/interceptor checks whether SHS needs to handle the request or pass it through.
Item is classified. SHS_ItemClassificationService resolves item/resource family from SHS types, additional type sidecars, override components, or compatibility aliases.
Slots are scanned. The resolver looks through managed storages and asks each slot whether it can accept the item/resource.
Candidate is scored. Exact prefab matches, policy slots, route priority, and family/category compatibility determine the winner.
Route executes. The inventory manager inserts or moves the item to the selected storage/slot.
Post-attach correction runs. Stored physics is suspended, placement transform is applied, and incompatible direct attachments can be bounced back through SHS routing.

3. Item types and families

The framework groups item types into semantic families. A slot does not merely ask “does this item have a common type?” It asks “does this item belong to the same family as this slot, and does it match the configured subcategory when required?”

Auto-routed range: Any type placed between SHS_AUTO_ROUTED_BEGIN and SHS_AUTO_ROUTED_END becomes part of the AUTO family, participates in SHS policy routing, and is hidden in first person. Do not assign the sentinel markers themselves to prefabs.

How type resolution works

SourcePurpose
Native / vanilla common typeBasic item classification when the prefab already has a reliable ECommonItemType.
m_SHS_AdditionalCommonItemTypesSidecar list for items that need one or more SHS routing tags without breaking their original type.
SHS_ItemTypeOverrideComponentPrefab-level override used when SHS should classify the item before trusting shared or external common types.
SHS_ExternalCompatibilityAliasesNormalizes external names such as GRS_RIFLE_AMMO into SHS-owned types such as SHS_RIFLE_AMMO.

Recommended item tagging rule

// For SHS-owned items:
Use SHS-owned ECommonItemType values directly or through m_SHS_AdditionalCommonItemTypes.

// For compatibility with external frameworks:
Prefer SHS_ItemTypeOverrideComponent or the additional sidecar list.

// For AUTO routed helmet/gear accessories:
Add the item type between SHS_AUTO_ROUTED_BEGIN and SHS_AUTO_ROUTED_END,
then configure the destination slot's m_SHS_MagCategory to that exact type.

4. Storage and slot setup

Primary slot attributes

m_SHS_MagCategoryCommonItemType/category gate. Set this to the SHS category the slot should accept, such as SHS_RIFLE_AMMO, SHS_RADIO, SHS_GRENADE, SHS_EAR_LEFT, etc.
m_SHS_SlotIDHuman-readable routing/placement ID. This must match SHS_MagPlacementData.PlacementID when visual offset/angles are used.
m_SHS_ItemPrefabExact prefab/resource rule. Leave blank for category fallback; fill it when one exact item must win this slot.
m_SHS_OccupancyGroupLogical physical-space lock. Slots with the same group block each other so variants cannot double-fill the same visual cell.

Policy-driven equipment storage

Use SHS_EquipmentStorageComponent for placards, belt cells, and visible item cells that need SHS slot rules. Enable its policy-driven flag for SHS-controlled cell routing.

Storage owner prefab
└─ SHS_EquipmentStorageComponent
   ├─ m_bSHS_PolicyDriven = true
   ├─ Slot 0: SHS_EquipmentStorageSlot
   │  ├─ m_SHS_MagCategory = SHS_RIFLE_AMMO
   │  ├─ m_SHS_SlotID = Placard_Cell_01
   │  └─ m_SHS_OccupancyGroup = Placard_Cell_01
   ├─ Slot 1: SHS_EquipmentStorageSlot
   │  ├─ m_SHS_MagCategory = SHS_PISTOL_AMMO
   │  ├─ m_SHS_SlotID = Belt_Cell_01
   │  └─ m_SHS_OccupancyGroup = Belt_Cell_01
   └─ Slot 2: SHS_EquipmentStorageSlot
      ├─ m_SHS_MagCategory = SHS_SMOKE_GRENADE
      ├─ m_SHS_SlotID = Chest_Left_01
      └─ m_SHS_OccupancyGroup = Chest_Left_01

Special policy slots

Slot classUse case
SHS_RadioStorageSlotRadio pouch or radio cell. Accepts radio components and/or SHS_RADIO typed items.
SHS_NadeStorageSlotExplosive / grenade-oriented slot for grenade pouch behavior.
SHS_PatchStorageSlotPatch/admin/ID slot accepting SHS_PATCH.

Avoid mixed-family fallback: A NONE slot category is treated as legacy ammo fallback. Do not use NONE for radios, smoke, grenades, patches, or AUTO gear. Give utility slots an explicit SHS category.

5. Visual placement offsets and rotations

Attach SHS_MagPlacementComponent to the storage owner when a stored item needs local visual adjustment after insertion.

Placement fields

FieldMeaning
PrefabExact item prefab/resource path. Highest-priority placement match. Leave blank for type or slot fallback.
MagTypeFallback common item type, such as SHS_RIFLE_AMMO.
PlacementIDMust match the slot’s m_SHS_SlotID.
OffsetLocal position offset applied after item attachment.
AnglesLocal rotation in Enfusion angle vector format.

Example placement setup

SHS_MagPlacementComponent.m_aPlacements:
  Entry 0:
    Prefab      = ""                         // optional exact prefab override
    MagType     = "SHS_RIFLE_AMMO"
    PlacementID = "Placard_Cell_01"
    Offset      = "0.015 0.020 -0.005"
    Angles      = "0 90 0"

Slot:
  m_SHS_MagCategory   = SHS_RIFLE_AMMO
  m_SHS_SlotID        = Placard_Cell_01
  m_SHS_OccupancyGroup= Placard_Cell_01
Why PlacementID matters: the item is first attached by the engine, then SHS looks up a placement using the attached item and the slot ID. If the placement ID and slot ID do not match exactly, no visual offset is applied.

6. Loadout area routing

Loadout areas are handled through a separate family registry. This lets SHS reason about equivalent areas without forcing every item to share the same engine-level canonical area.

How to add a new routable area

// 1. Define the area class.
class SHS_My_New_Area : LoadoutAreaType {};

// 2. Add a family if no existing family fits.
enum SHS_EAreaFamily
{
    // existing values...
    MY_NEW_FAMILY
}

// 3. Register the area-to-family mapping.
new SHS_AreaRoutingMapping(SHS_My_New_Area, SHS_EAreaFamily.MY_NEW_FAMILY)
Important: area-family routing does not override the engine’s equip acceptance by itself. If a cloth prefab must satisfy extra SHS-side areas, add SHS_ClothAreaExtensionComponent and list the additional area types there.

7. How-to recipes

Recipe A — Create a rifle-mag placard cell

On the placard/storage owner, add or use SHS_EquipmentStorageComponent.
Set m_bSHS_PolicyDriven to true.
Add an SHS_EquipmentStorageSlot.
Set m_SHS_MagCategory to SHS_RIFLE_AMMO.
Set m_SHS_SlotID to a unique visual ID, for example Placard_Cell_01.
Set m_SHS_OccupancyGroup to the same physical group, for example Placard_Cell_01.
Optional: add a placement entry with PlacementID = Placard_Cell_01.

Recipe B — Create multiple visual variants for one physical cell

Use different slot IDs but the same occupancy group. This lets STANAG, AK, DMR, or MG variants have different visuals while still occupying one physical cell.

Slot A:
  m_SHS_MagCategory    = SHS_RIFLE_AMMO
  m_SHS_SlotID         = STANAG_01
  m_SHS_OccupancyGroup = Placard_Cell_01

Slot B:
  m_SHS_MagCategory    = SHS_AK_AMMO
  m_SHS_SlotID         = AK_01
  m_SHS_OccupancyGroup = Placard_Cell_01

Slot C:
  m_SHS_MagCategory    = SHS_MARKSMAN_AMMO
  m_SHS_SlotID         = DMR_01
  m_SHS_OccupancyGroup = Placard_Cell_01

Recipe C — Add a new ammo category

Add a new SHS-owned value to ECommonItemType, such as SHS_SHOTGUN_AMMO.
Update SHS_ItemClassificationService.SHS_IsAmmoCategory() so the new type resolves to the AMMO family.
Tag the item prefab using its common type, additional type sidecar, or SHS_ItemTypeOverrideComponent.
Create one or more slots with m_SHS_MagCategory = SHS_SHOTGUN_AMMO.
Add placement entries where needed.

Recipe D — Add an auto-routed headset, NVG, or rail accessory

Add the new ECommonItemType between SHS_AUTO_ROUTED_BEGIN and SHS_AUTO_ROUTED_END.
Tag the item prefab with that new type or add it to m_SHS_AdditionalCommonItemTypes.
Create a destination slot with m_SHS_MagCategory set to that exact type.
Use a position-specific type for position-specific gear, for example left ear versus right ear.
Do not manually add the type to first-person hide logic unless it is outside the sentinel range.

Recipe E — Make a radio pouch

Use a pouch/storage component that contains a policy slot.
Add SHS_RadioStorageSlot as the storage slot.
Tag radio items with SHS_RADIO or ensure they have a radio component detected by classification.
Do not rely on a generic NONE mag slot for radios.

Recipe F — Add external compatibility without taking dependency on another framework

Add string normalization inside SHS_ExternalCompatibilityAliases.NormalizeExternalItemTypeName().
Map the external name to an SHS-owned category, for example External_RIFLE_AMMO to SHS_RIFLE_AMMO.
Keep SHS routing pointed at SHS-owned categories.
Avoid importing or directly requiring external framework classes unless runtime-detected by string.

8. Troubleshooting

ProblemLikely cause and fix
Item goes to cell 1 or wrong cellThe target slot may have m_SHS_MagCategory = NONE, missing policy-driven storage, or missing item type tagging. Set a specific SHS category and verify the item resolves to the same family/category.
Radio / smoke / grenade goes into mag slotA generic fallback slot is being treated as ammo. Give utility slots explicit categories and use specialized policy slots where applicable.
Two variants fill the same physical locationThe visual variant slots do not share the same m_SHS_OccupancyGroup. Assign the same group to all mutually-exclusive variants.
Item accepts in Workbench but routes wrong in gameDirect attachment paths may bypass the normal insert route. Check family rebound logs if enabled and confirm IsItemCompatible would accept the item.
Offset/rotation does not applyThe storage owner may be missing SHS_MagPlacementComponent, or PlacementID does not exactly match the slot’s m_SHS_SlotID.
Auto-routed item does not hide in first personThe item type may not be between the auto-routed sentinel markers, or the prefab is missing the additional/common type tag. For native types, add to SHS_FP_HIDE_EXTRA_TYPES.
External framework item is not recognizedAdd or correct a normalization entry in SHS_ExternalCompatibilityAliases, then verify the final name maps to a valid SHS common type.
Cloth item should count as another area but does notArea families do not change engine equip acceptance. Add SHS_ClothAreaExtensionComponent with the additional SHS area types.

9. Authoring checklist

Before testing a new modular item

  • Item prefab has a valid SHS common type, additional type, or override component.
  • Destination storage uses the correct SHS storage/component class.
  • Policy-driven storage is enabled where required.
  • Slot category matches the intended item family and subcategory.
  • Slot ID is unique enough for diagnostics and placement lookup.
  • Occupancy group is shared by all variants of the same physical space.
  • Placement component exists on the storage owner if offsets/angles are needed.
  • PlacementID exactly matches the slot ID.
  • External names are normalized to SHS-owned categories.
  • Auto-routed gear is inside the sentinel range and has position-specific slot categories where needed.

Slot IDs:

  Placard_Cell_02
  Belt_Cell_01
  Chest_Left_01
  Ear_Left_01
  Helmet_Rail_Right_01

  SHS_RIFLE_AMMO
  SHS_PISTOL_AMMO
  SHS_RADIO
  SHS_EAR_LEFT
  SHS_RailLiteRight
Best practice: keep routing category, slot ID, occupancy group, and placement ID intentionally separate. Category answers “what item is allowed?” Slot ID answers “which visual slot is this?” Occupancy group answers “what physical space is occupied?” Placement ID answers “which transform should be applied?”
